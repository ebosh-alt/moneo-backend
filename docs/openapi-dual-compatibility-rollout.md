# Стратегия поэтапной совместимости данных (dual-read / dual-write)

## 1. Цель

Обеспечить безопасный rollout при временной поддержке нескольких форматов данных, без регрессий API и без потери данных.

Документ дополняет:

- `docs/openapi-format-compatibility-matrix.md` (правила форматов полей);
- OpenAPI-first процесс (`api/openapi.yaml` + generated transport).

## 2. Когда включать dual-режим

Dual-read/dual-write включается только при наличии хотя бы одного условия:

- в хранилище уже есть записи в legacy-формате, несовместимом с новым чтением;
- внешние интеграции продолжают отправлять legacy-формат;
- перенос делается поэтапно между сервисами и одновременно живут старые и новые контракты.

Если данные уже нормализованы и все клиенты обновлены, dual-режим не включается.

## 3. Фазы rollout

## Фаза 0. Подготовка

- Зафиксировать формат-матрицу и edge-cases.
- Добавить метрики и алерты по parse/validation ошибкам.
- Подготовить feature-флаги совместимости.

Критерий выхода:

- есть контрактные тесты happy/negative;
- есть observability по доле fallback/read-repair ошибок.

## Фаза 1. Dual-read + single-write (новый формат)

- Чтение: сначала новый формат, при неуспехе fallback на legacy.
- Запись: только в новый формат.
- При успешном fallback чтении делать read-repair (перезапись в новый формат) асинхронно или в безопасной синхронной ветке.

Критерий выхода:

- fallback-rate стабильно снижается;
- нет роста 4xx/5xx и бизнес-инцидентов.

## Фаза 2. Dual-read + dual-write (только если реально нужно)

- Запись: одновременно в новый и legacy формат.
- Источник истины для чтения: новый формат (legacy только fallback).
- При расхождении write-результатов: ошибка операции + событие в мониторинг (без silent-ignore).

Критерий выхода:

- write parity подтвержден;
- доля write mismatch близка к нулю и объяснима.

## Фаза 3. Read new only (legacy fallback выключен)

- Чтение только из нового формата.
- Legacy write может оставаться на короткий grace-период, если нужен безопасный откат.

Критерий выхода:

- стабильность на проде в течение согласованного окна наблюдения.

## Фаза 4. Полное выключение совместимости

- Выключить legacy write.
- Удалить fallback/read-repair код.
- Оставить только новый контракт и инварианты.

Критерий выхода:

- код и процесс не содержат legacy-веток;
- документация и тесты обновлены.

## 4. Порядок чтения/записи при смешанных данных

## Чтение (при dual-read)

1. Пробуем читать новый формат.
2. Если запись не найдена или не парсится по новому правилу, читаем legacy.
3. Если legacy прочитан успешно, нормализуем в доменный вид и отмечаем событие fallback.
4. По политике read-repair перезаписываем в новый формат.

Важно: при конфликте данных приоритет у нового формата, если не объявлен инцидент качества данных.

## Запись (single-write new)

1. Валидируем вход только по текущему OpenAPI-контракту.
2. Пишем в новый формат.
3. Ошибка записи возвращается клиенту стандартным `error envelope`.

## Запись (dual-write)

1. Валидируем один раз на входе.
2. Пишем в новый формат (основной путь).
3. Пишем в legacy формат (теневой путь совместимости).
4. Фиксируем статус обеих операций для метрик parity.

Важно: бизнес-логика и транзакционные правила остаются в `internal/app`, не переносятся в transport/infra.

## 5. Feature-флаги и условия переключения

Рекомендуемые флаги (имена могут быть адаптированы под конфиг проекта):

- `compat.dual_read_enabled`
- `compat.dual_write_enabled`
- `compat.read_repair_enabled`
- `compat.legacy_write_enabled`

Правила:

- включаем/выключаем только через rollout-окно;
- меняем не более одного ключевого флага за шаг;
- каждый шаг сопровождается мониторингом минимум N часов (значение N согласуется командой заранее).

## 6. Метрики и контроль рисков

Обязательные метрики:

- `compat_read_fallback_total` и доля fallback от всех read;
- `compat_read_repair_total` и `compat_read_repair_failed_total`;
- `compat_dual_write_mismatch_total`;
- `validation_error_rate` и `5xx_rate` по endpoint-группам.

Стоп-условия rollout:

- резкий рост `5xx`;
- рост mismatch/fallback выше согласованного порога;
- подтвержденные пользовательские регрессии по money/date/type/status.

## 7. План отката

## Мягкий откат (первый шаг)

1. Выключить `dual_write_enabled`.
2. Вернуть `dual_read_enabled` (если был выключен).
3. Оставить новый формат основным, но с fallback на legacy.

## Жесткий откат (инцидент)

1. Отключить изменения последней волны rollout.
2. Временно включить `legacy_write_enabled`, если требуется восстановление совместимости.
3. Зафиксировать инцидент и провести data diff между new/legacy слоем.

После отката обязательно:

- заморозка следующих фаз до RCA;
- обновление матрицы и тестов по результатам RCA.

## 8. Правила принятия решения о переходе между фазами

Переход разрешен только если одновременно выполнено:

- все критичные алерты зеленые;
- нет открытых P1/P2 багов миграции;
- есть подтверждение от backend + QA + владельца продукта.

## 9. Минимальный checklist перед закрытием задачи

- есть согласованный rollout-план по фазам;
- зафиксированы включатели/выключатели совместимости;
- определен порядок чтения/записи для mixed data;
- описан пошаговый rollback;
- метрики и пороги инцидентов задокументированы.
